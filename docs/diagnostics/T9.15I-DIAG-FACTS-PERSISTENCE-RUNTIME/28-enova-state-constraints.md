## Impacto no fix T9.15J

### Fix original proposto (file 27) — NÃO FUNCIONA diretamente

Usar `wa_id` como chave de `supabaseUpsert` em `enova_state`:
```typescript
const row: EnovaStateRow = {
  wa_id: "554185260518",
  last_context: JSON.stringify(facts),
  updated_at: ...,
};
await supabaseUpsert(cfg, 'enova_state', row);
```

PostgREST `Prefer: resolution=merge-duplicates` upserta pela **PK** (`id`) ou por uma **UNIQUE constraint**.
`wa_id` não é PK nem UNIQUE em `enova_state` → upsert por `wa_id` inseriria linha duplicada em vez de atualizar.

### Fix alternativo A — SELECT by wa_id, depois UPDATE by id (mais seguro)

```typescript
// 1. SELECT: buscar id do row existente por wa_id
const existing = await supabaseSelect('enova_state', { filters: { wa_id: `eq.${wa_id}` }, limit: 1 });
if (existing.rows[0]?.id) {
  // 2. UPDATE: atualizar last_context usando id como chave
  await supabaseUpsert('enova_state', { id: existing.rows[0].id, last_context, updated_at });
} else {
  // 3. Linha não existe ainda — não criar (sem UUID de lead válido)
}
```

Vantagem: sem risco de duplicata. Desvantagem: 2 round-trips Supabase por turno.

### Fix alternativo B — Verificar se wa_id tem UNIQUE no Supabase real (via SQL direto)

Antes de qualquer fix, confirmar no banco:
```sql
SELECT constraint_name, constraint_type
FROM information_schema.table_constraints
WHERE table_name = 'enova_state';
```

Se `wa_id` tiver UNIQUE → upsert por `wa_id` funciona diretamente.
Se não tiver → Fix A ou adicionar UNIQUE constraint via migration.

### Fix alternativo C — Passar wa_id separado para read/write (sem depender de lead_id)

Em `canary-pipeline.ts`, o `event.wa_id` está acessível via contexto. Passar explicitamente
como argumento separado em `readLeadAccumulatedFacts(supabaseCfg, wa_id)` e `writeLeadAccumulatedFacts(supabaseCfg, wa_id, facts)`.

Porém, a questão de constraint continua sendo pré-requisito para o upsert funcionar.
