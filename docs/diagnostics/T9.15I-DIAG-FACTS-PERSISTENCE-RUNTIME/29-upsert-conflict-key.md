## Função supabaseUpsert completa (src/supabase/client.ts:282–355)

```typescript
/**
 * Upsert (insert-or-update) via PostgREST resolution=merge-duplicates.
 * Usado em T9.12 para escrita real de crm_lead_meta e enova_state.
 * Nunca lança em erro de rede — encapsula em result.ok=false.
 * Nunca expõe serviceRoleKey em mensagem de erro.
 */
export async function supabaseUpsert<T>(
  cfg: SupabaseConfig,
  table: string,
  row: T,
): Promise<SupabaseQueryResult<T>> {
  const url = `${cfg.url.replace(/\/$/, '')}/rest/v1/${encodeURIComponent(table)}`;

  let response: Response;
  try {
    response = await fetch(url, {
      method: 'POST',
      headers: {
        'content-type': 'application/json',
        accept: 'application/json',
        prefer: 'resolution=merge-duplicates,return=representation',   // ← CHAVE
        apikey: cfg.serviceRoleKey,
        authorization: `Bearer ${cfg.serviceRoleKey}`,
      },
      body: JSON.stringify(row),
    });
  } catch (e) {
    return { ok: false, rows: [], total: 0, error: '...', http_status: null };
  }

  // ... parse response, return ok/error ...
}
```

## Como PostgREST resolve `resolution=merge-duplicates`

### Comportamento oficial PostgREST

```
Prefer: resolution=merge-duplicates
```

PostgREST traduz para SQL:
```sql
INSERT INTO <table> (<cols>) VALUES (<vals>)
ON CONFLICT (<conflict_target>) DO UPDATE SET ...
```

**`<conflict_target>`** é determinado por PostgREST automaticamente a partir de:
1. **PK da tabela** (sempre)
2. **UNIQUE constraints** (cada constraint vira um conflict target alternativo)

**Não existe `on_conflict` column explícito** nesta implementação — a função não aceita
parâmetro de conflict target. PostgREST usa os constraints do schema do Postgres.

### Implicação para enova_state

`enova_state` schema real:
- **PK**: `id` (UUID autogenerated)
- **`lead_id`**: UUID, FK constraint (`enova_state_lead_id_fkey`) — FK **não** implica UNIQUE
- **`wa_id`**: TEXT — UNIQUE constraint: **desconhecida** (não documentada nos diagnósticos)

**Se o payload NÃO inclui `id` (PK):**
- PostgREST não encontra conflito pelo PK → `ON CONFLICT (id) DO UPDATE` nunca dispara
- Se `wa_id` tiver UNIQUE → `ON CONFLICT (wa_id) DO UPDATE` atualiza linha existente ✅
- Se `wa_id` NÃO tiver UNIQUE → nenhum conflito detectado → **INSERT nova linha** ❌

## Análise do risco para T9.15J

### writeLeadAccumulatedFacts ANTES (T9.15H, com lead_id)

```typescript
const row: EnovaStateRow = {
  lead_id,          // UUID (se válido) ou wa_id (em PROD = "554185260518")
  last_context: JSON.stringify(facts),
  updated_at: new Date().toISOString(),
};
await supabaseUpsert(cfg, 'enova_state', row);
```

Em PROD: `lead_id = "554185260518"` → `pg_code=22P02` → falha antes de chegar ao ON CONFLICT.
Nunca havia inserido nada em `enova_state.last_context`.

### writeLeadAccumulatedFacts DEPOIS (T9.15J, com wa_id)

```typescript
const row: EnovaStateRow = {
  wa_id,            // "554185260518" — TEXT, tipo correto
  last_context: JSON.stringify(facts),
  updated_at: new Date().toISOString(),
};
await supabaseUpsert(cfg, 'enova_state', row);
```

**Cenário A — wa_id tem UNIQUE constraint:**
```sql
INSERT INTO enova_state (wa_id, last_context, updated_at) VALUES (...)
ON CONFLICT (wa_id) DO UPDATE SET last_context=..., updated_at=...
```
→ Primeira vez: INSERT. Turno 2+: UPDATE. ✅ Comportamento correto.

**Cenário B — wa_id NÃO tem UNIQUE constraint:**
```sql
INSERT INTO enova_state (wa_id, last_context, updated_at) VALUES (...)
-- ON CONFLICT nunca dispara (wa_id não é PK nem UNIQUE)
```
→ Cada turno INSERE nova linha. `enova_state` acumula duplicatas. ⚠️

**Cenário B — read após duplicatas:**
```typescript
const result = await supabaseSelect('enova_state', {
  filters: { wa_id: 'eq.554185260518' },
  limit: 1,
});
```
PostgREST sem `order` → retorna qualquer uma das linhas (não determinístico).
Pode retornar a mais antiga (vazia) em vez da mais recente. ⚠️

## Como o upsertLeadState existente funciona em PROD

O `SupabaseCrmBackend.update` para `enova_state` provavelmente inclui `lead_id` no payload
(quando tem UUID válido de lead). Se `enova_state.lead_id` tiver UNIQUE constraint,
o merge-duplicates funciona por `lead_id`. Mas em PROD `lead_id = wa_id` → pg_code=22P02
(mesmo bug T9.15J, porém no upsertLeadState — investigar separadamente).

## Conclusão e recomendação

### Para T9.15J funcionar corretamente em PROD, há dois caminhos:

**Caminho 1 — Confirmar que wa_id tem UNIQUE (preferível)**

Query no Supabase real (Vasques):
```sql
SELECT
  tc.constraint_name,
  tc.constraint_type,
  kcu.column_name
FROM information_schema.table_constraints tc
JOIN information_schema.key_column_usage kcu
  ON tc.constraint_name = kcu.constraint_name
WHERE tc.table_name = 'enova_state'
ORDER BY tc.constraint_type;
```

Se retornar `UNIQUE` em `wa_id` → T9.15J funciona como esperado. ✅
Se não retornar → PR #238 precisa de ajuste antes do merge.

**Caminho 2 — Fix alternativo se wa_id não tiver UNIQUE**

Substituir `supabaseUpsert` por `supabaseSelect` + update condicional:

```typescript
// 1. Busca row existente por wa_id
const existing = await supabaseSelect('enova_state', { filters: { wa_id: `eq.${wa_id}` }, limit: 1 });
if (existing.ok && existing.rows.length > 0 && existing.rows[0].id) {
  // 2. Upsert por id (PK garantida)
  await supabaseUpsert('enova_state', { id: existing.rows[0].id, last_context, updated_at });
} else {
  // 3. Sem row existente — não criar (sem uuid de lead válido)
  return { ok: false, error: 'no_existing_row_for_wa_id' };
}
```

Custo: +1 round-trip Supabase por turno de escrita. Sem risco de duplicatas.

**Caminho 3 — Adicionar UNIQUE constraint via migration**

```sql
ALTER TABLE enova_state ADD CONSTRAINT enova_state_wa_id_unique UNIQUE (wa_id);
```
