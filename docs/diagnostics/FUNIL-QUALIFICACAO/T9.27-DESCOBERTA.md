## 2. Estrutura atual de text-extractor.ts

### extractRenda (linhas 563-588)

Aceita:
- `"4 mil"`, `"3,5 mil"` → regex `/(\d+(?:[,.]\d+)?)\s*mil/`
- `"R$ 3.500"`, `"R$3500"` → regex `/r\$\s*([\d.,]+)/`
- `"renda de 4000"`, `"ganho 4000"`, `"recebo 4000"`, `"salario de 3500"` → regex keyword-preceded

- `"2500"` (número puro sem prefixo) → null
- `"três mil"` (extenso por escrito) → null

## 3. Bugs confirmados

### BUG 1 — renda_principal não aceita número puro
`extractRenda("2500")` → null. Cenários C1, C3, C7, C8 do E2E enviam renda como número puro.
Fix: criar `parseRendaFlexivel` com fallback para número puro, extensos e notação k.

### BUG 2 — estado_civil_p3 não capturado em discovery
Quando stage ainda é `discovery` e pendingObjective é `coletar_estado_civil_p3`, não existe
bloco em `extractDiscovery`. C3 (estado_civil_p3 da mãe) pode cair nesse gap.
Fix: adicionar bloco em extractDiscovery + guard no bloco estado_civil existente.

### BUG 3 — rnm_valido: ordenação de negativas poderia ser mais robusta
A ordenação atual coloca "tem validade", "prazo determinado" junto com "nao tenho" e "nao"
no mesmo if. Ordenação mais granular (específico → genérico) previne futuros falsos positivos.

### BUG 4 — alternativa_rnm: sem valor 'sem_familiar_brasileiro'
E2E C6 envia "Nao tenho familiar brasileiro" e valida `alternativa_rnm='sem_familiar_brasileiro'`.
Código atual emite `'sem_alternativa'`. Falta granularidade + semantic pendingObjective.
